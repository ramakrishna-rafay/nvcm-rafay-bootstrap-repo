Bootstrap load order — matches LoadBootstrapData.run() in jobs/load_bootstrap_data.py.
Each item loads after its dependencies. All loaders are idempotent (update_or_create):
re-running reconciles existing objects to the YAML. Deletes are NOT pruned (manual cleanup).

 1. manufacturers.yaml           (no dependencies)
 2. tenants.yaml                 (no dependencies)
 3. location_types.yaml          (no dependencies; self-referential chain
                                  Provider -> Region -> Site -> Building -> Room)
 4. namespaces.yaml              (no dependencies)
 5. statuses.yaml                (no dependencies)
 6. locations.yaml               (depends on location_types, statuses, tenants)
 7. rack_groups.yaml             (depends on locations)
 8. racks.yaml                   (depends on locations, statuses, rack_groups; optional tenant, role)
 9. roles.yaml                   (no dependencies)
10. tags.yaml                    (no dependencies)
11. custom_fields.yaml           (no dependencies)
12. platforms.yaml               (depends on manufacturers)
13. device_types/                (depends on manufacturers). Also loads each type's interface
                                  templates from the YAML `interfaces:` block (name/type/mgmt_only/
                                  mtu), so devices created afterwards get the full port set.
14. relationships.yaml           (no dependencies)
15. config_context_schemas.yaml  (no dependencies)          [optional - no-op if file absent]
16. config_contexts.yaml         (depends on config_context_schemas; roles/locations if scoped)
                                                            [optional - no-op if file absent]
17. device_placements.yaml       (rack elevations: device -> rack/position/face). Runs LAST — needs
                                  the DEVICES to already exist (created by the fabric/server DesignJobs),
                                  so run/re-run this after them; a missing device is skipped, not errored.
                                  Entries with `skip: true` are UI-managed and ignored (e.g. the spines).
18. (interface backfill)         Not a file — runs after device_placements. Adds to each EXISTING
                                  device any interface its device-type template defines but the device
                                  lacks (templates only auto-instantiate at device-creation time), and
                                  fills a missing MTU from the template. Idempotent; ports already
                                  present keep their attributes. New devices get their ports at create.
